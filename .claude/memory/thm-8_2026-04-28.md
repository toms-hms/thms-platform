**Key decisions:**
- `category` column dropped from `Contractor` table; categories live in `ContractorCategory` (contractorId + category composite PK)
- `ContractorZipCode` has its own `id` PK (not composite) to allow future dedup/ordering
- `ContractorManager.create/update` takes explicit `categories` and `zipCodes` args, not embedded in the data object
- Update replaces categories/zipCodes in full (delete-all + re-insert) when those args are provided; omitting them leaves existing values unchanged
- Migration required: run `make migrate` to generate and apply the schema diff
